Game Engine

Unity

The engine behind half the world's games — from Pokémon GO to Hollow Knight to Genshin Impact. Here's how it works and how to get started.

What Is Unity?

Unity is a real-time development platform — a game engine, scene editor, and build toolchain bundled together. You build a level by placing objects in a 3D viewport, attaching behaviour scripts written in C#, and hitting Play to run the game instantly inside the editor. When you're done, Unity can export to over 25 platforms: iOS, Android, Windows, Mac, Linux, PlayStation, Xbox, Nintendo Switch, and WebGL — all from one project, with minimal platform-specific code.

It's not just for games, either. Unity powers car configurators, architecture walkthroughs, medical training simulations, and film VFX. Weta Digital used it for previsualization on major productions. But at its core, Unity was built by game developers for game developers — and that's where it shines most clearly.

Scale
71% of the top 1,000 mobile games are built with Unity. On Steam, roughly half of all released titles use it. More games run on Unity than on any other engine — by a wide margin.

History

Unity's story starts not with a grand vision but with a failed game. In 2004, three developers in Copenhagen — David Helgason (CEO), Nicholas Francis (CCO), and Joachim Ante (CTO) — founded a company called Over the Edge Entertainment. Their goal was to make games.

In 2005 they finished their debut title, GooBall, a physics puzzle game for Mac. It sold almost nothing. But the engine they'd built to make it was genuinely impressive: drag-and-drop assets, instant play-in-editor, a visual workflow they'd inspired themselves to build after admiring Apple's Final Cut Pro. They pivoted immediately and decided to sell the engine itself.

How It Works

Unity is built around one central idea: everything in your game is a GameObject. A player character, a camera, a lamp, a sound trigger, a piece of invisible collision geometry — all GameObjects. They are empty containers. Their behaviour comes entirely from Components you attach to them.

The Component model

This architecture replaces the deep inheritance hierarchies common in older engines. Instead of subclassing a CollidingMovingRenderedEntity, you attach a Rigidbody, a MeshRenderer, a Collider, and your own PlayerController script. Each component does one job. GameObjects compose them freely. Adding a new ability means writing a new component and attaching it — not refactoring a class tree.

Every GameObject automatically carries a Transform component that cannot be removed. It stores position, rotation, and scale in 3D space, and the parent-child relationship that lets you build nested object hierarchies (a hand attached to an arm attached to a body, all moving together).

Scenes

A Scene is a discrete level or screen — the main menu, world 1-1, the boss arena. Your game can contain many scenes and load them additively or sequentially. Unity's game loop runs continuously while a scene is active: process input → run physics → call each script's Update() → render the frame → repeat. At 60 fps this ticks 60 times per second. You never write the loop — Unity calls your methods into it.

C# Scripting

Scripts are C# classes that extend MonoBehaviour, Unity's base class for anything that lives in a scene. Two lifecycle methods are enough to get started:

using UnityEngine; public class PlayerController : MonoBehaviour { public float speed = 5f; private Rigidbody rb; // Called once, before the first frame renders void Start() { rb = GetComponent<Rigidbody>(); } // Called every frame — ~60× per second at 60 fps void Update() { float h = Input.GetAxis("Horizontal"); float v = Input.GetAxis("Vertical"); Vector3 move = new Vector3(h, 0f, v) * speed; rb.velocity = move; } }

That's a complete, working player controller — about 20 lines. Notice the pattern: Start() to initialise once, Update() to tick every frame. Unity calls these automatically; you write the what, not the when.

Surprising detail
Unity compiles your C# to an intermediate language, then uses a technique called IL2CPP to convert that to C++ source, which is then compiled to native machine code for each target platform. The result is faster runtime performance and makes your game logic much harder to decompile. A consequence: stack traces on device can look alien compared to what you see in the editor.

The event execution order

Unity calls MonoBehaviour methods in a specific order. Knowing this order prevents a whole class of bugs:

// Initialisation (once per object lifetime) Awake() // even if the component is disabled — use for self-setup OnEnable() // every time the object is activated Start() // before the first frame, if enabled — use for cross-object wiring // Per-frame (variable rate) Update() // input, game logic, timers LateUpdate() // after ALL Update()s — camera follow goes here // Fixed timestep (default 50 Hz — decoupled from framerate) FixedUpdate() // all physics forces and velocity changes go here, NOT Update()

The single most common beginner mistake: putting physics forces in Update(). Because Update() runs at the display framerate (variable), physics driven from it becomes framerate-dependent — running faster on a 144 Hz monitor than a 30 Hz phone. FixedUpdate() runs at a fixed 50 Hz by default, completely decoupled from rendering.

Prefabs

A Prefab is a saved template of a configured GameObject and all its components. Drag a prefab into a scene to spawn an instance. Change the prefab template and every instance updates automatically. This is how you create 200 identical enemies without configuring 200 separate objects — and how you safely update them all in one edit.

Render Pipelines

One of Unity's most confusing aspects for newcomers: it has three distinct rendering backends. You choose one per project at creation time. The choice affects which shaders work, which post-processing effects are available, and how lighting behaves. Switching pipelines later is painful — it requires converting all materials.

Built-In Pipeline

Legacy

Unity's original renderer. Supports almost everything but is not extensible or particularly performant. Still used by thousands of older tutorials online. Do not start new projects with it — you'll hit walls quickly.

Universal Render Pipeline

Recommended

Unity's current default for new projects. Fast across mobile, console, and PC. Built on the Scriptable Render Pipeline — you can customise the rendering pass order in code. Most third-party assets now support URP. This is the right choice for the vast majority of new games.

High Definition Render Pipeline

High-Fidelity

Physically-based lighting, volumetric fog, ray tracing, subsurface scattering. Requires powerful hardware — unsuitable for mobile. Used for cinematic-quality PC and console games. Beautiful results, steep learning curve, limited asset store compatibility.

Classic gotcha
If you find a Unity shader or post-effect tutorial online, check which pipeline it targets. A shader written for HDRP is incompatible with URP and will show as a pink error material. This trips up nearly every beginner at least once.

Getting Started

Zero to running project takes about 15 minutes. Here's the direct path:

  1. Install Unity Hub

    Unity Hub is the launcher and version manager — download it from unity.com. It lets you install multiple Unity versions side-by-side. Always install the latest LTS (Long Term Support) release for new projects. LTS versions receive bug fixes for two years; non-LTS releases are stable but receive fewer patches.

  2. Create a project from a template

    In Unity Hub click New Project. Choose 3D (URP) for a 3D game or 2D (URP) for a side-scroller or top-down game. The template pre-configures the render pipeline and asset import settings for you. Do not choose the legacy "3D (Built-In)" template.

  3. Learn the five core panels

    Scene — your 3D editor viewport. Hierarchy — all GameObjects in the active scene. Project — your files on disk. Inspector — properties of whatever's selected. Game — what the player's camera sees. Every tutorial references these five constantly. Recognise them early.

  4. Write your first C# script

    In the Project panel: right-click → Create → C# Script. Name it (no spaces). Double-click to open in Visual Studio or Rider. The file opens with Start() and Update() already stubbed. In the Hierarchy, select a GameObject, then drag your script file onto it in the Inspector. Hit Play.

  5. Do the official beginner tutorials

    Unity Learn (learn.unity.com) is free and genuinely well-made. The Creator Kit: Beginner Code project hands you a complete action RPG and teaches you to extend it with scripts. Do this before anything else — it teaches the editor, the Hierarchy, Components, and C# fundamentals simultaneously in a real context.

Editor tip
For your IDE, consider JetBrains Rider over Visual Studio. It has Unity-specific intelligence: it knows which methods are called by the engine (not just your code), warns about expensive calls inside Update(), and autocompletes the full Unity API with documentation inline. A free 30-day trial is enough to know if it suits you.

Genshin Impact is arguably the most technically impressive Unity game ever shipped: a fully open-world action RPG running in real-time on iOS, Android, PC, and PlayStation simultaneously, with hundreds of millions of downloads. It dismantled the assumption that Unity had a low ceiling for AAA-scale ambition.

Cuphead's 1930s rubber-hose animation — which looks like it was cel-animated on actual film — runs inside Unity's 2D animation system with custom shaders that mimic film grain, projector flicker, and colour bleed. The visual style required vastly more work than the code underneath it.

Surprising detail
Among Us was released in 2018 to almost zero audience. It sat dormant for two years. Then, in 2020, a handful of large Twitch streamers picked it up during lockdowns, and it exploded to over 500 million active players. The Unity codebase hadn't been optimised for that scale — the developers spent months of emergency patching to keep servers alive. The engine held; the backend infrastructure didn't.

Unity vs Unreal Engine

The two dominant engines are often framed as rivals. They're better understood as different tools for different jobs — each genuinely better than the other in specific contexts.

Area Unity Unreal Engine 5
Primary language C# — approachable, excellent tooling C++ and Blueprints (visual scripting)
Mobile games Dominant — 71% of top 1K mobile titles Possible but uncommon
AAA visuals Very good via HDRP Industry-leading — Nanite, Lumen, ray tracing
Asset marketplace Massive — 70,000+ Unity Asset Store items Fab marketplace — growing fast since 2024
Learning curve Gentler — especially for C# newcomers Steeper — more concepts to understand upfront
2D games First-class 2D pipeline built-in Possible, but feels like a workaround
Indie presence Enormous — most indie games use Unity Growing rapidly, historically AAA-only
Royalties 5% on revenue above $200K/year (Unity Pro) 5% on revenue above $1M lifetime

Honest guidance: if you're making a mobile game, a 2D game, or learning game development for the first time — start with Unity. If your target is a photorealistic first-person PC or console game with cinematic lighting — consider Unreal. There's no universally wrong answer, but there is a better fit for each project type.

The 2023 Runtime Fee Crisis

No honest overview of Unity omits this. It's one of the most spectacular trust failures in the modern games industry — and understanding it matters if you're thinking about building a business on this engine.

What was announced

On September 12, 2023, Unity announced a new Runtime Fee: developers would be charged $0.20 per install for any game that had crossed $200K in revenue AND 200K lifetime installs. The policy was retroactive — it applied to games already shipped, built under entirely different contractual terms.

The fee was per-install, not per-sale. A game downloaded and deleted repeatedly would accrue fees with zero corresponding revenue. Unity had no technical means to distinguish genuine purchases from piracy, reinstalls, or game pass includes — meaning developers would be billed for copies they never earned a penny from.

The developer response

The backlash was immediate and unusually unified. Innersloth (Among Us) announced they were evaluating migrating off Unity. Facepunch (Rust) declared they would never make another Unity game. Massive Monster threatened to delete their game entirely rather than pay retroactive fees. Within days, dozens of studios had published migration timelines to Godot and Unreal Engine.

Compounding the problem: Unity CEO John Riccitiello had previously, in an unrelated interview, called developers who didn't think about monetisation from day one "f***ing idiots." That quote resurfaced immediately and destroyed any remaining goodwill.

Outcome

Unity reversed course within weeks. The Runtime Fee was formally cancelled. Riccitiello resigned. The policy was replaced with traditional subscription price increases — painful, but comprehensible. The long-term damage, however, was real: trust eroded substantially. Godot (the open-source alternative) saw a 300% spike in contributors and downloads. Unity's Steam market share has declined year-on-year since 2023, with Unreal gaining ground.

The structural lesson
The Runtime Fee is now a case study in how not to change platform pricing. The critical mistake wasn't the fee itself — it was the retroactive application to games built under different terms, and the per-install trigger that made future costs impossible to model. Developers had built businesses on the assumption that the terms they shipped under would remain. That assumption turned out to be wrong.

Quiz

Test what you've picked up. Answer as you go, then hit Check Answers for your score.

1. What is the core building block of a Unity scene?
GameObjects are empty containers that hold components — including scripts, renderers, colliders, and physics bodies. A Transform is a component on a GameObject. Prefabs are saved templates of GameObjects.
2. Where should physics forces and velocity changes be applied in a Unity script?
FixedUpdate() runs on a fixed physics timestep (default 50 Hz), decoupled from the rendering framerate. Driving physics from Update() creates framerate-dependent behaviour — your game runs faster at 144 Hz than at 30 Hz.
3. Which render pipeline is Unity's recommended default for new projects in 2024–2026?
URP is Unity's current recommended default — it performs well on mobile through to high-end PC. HDRP is for high-fidelity PC/console games only. The Built-In pipeline is legacy. SRP is the framework both URP and HDRP are built on — it's not a pipeline you'd select directly.
4. Where and when did Unity 1.0 publicly launch?
The three Copenhagen founders demoed Unity 1.0 at Apple's Worldwide Developers Conference in June 2005 — initially as a Mac-only engine. The reaction from Mac developers confirmed their pivot from making games to selling the engine.
5. What was the fundamental structural flaw in the 2023 Unity Runtime Fee that made it unworkable for developers?
The fee triggered on installs, not revenue — meaning pirated copies, reinstalls, and game pass downloads all counted. It applied retroactively to games already shipped under different terms. Unity also had no technical way to distinguish legitimate installs from ghost ones, making future costs impossible to forecast.
6. Approximately what share of the top 1,000 mobile games are built with Unity?
71% of the top 1,000 mobile games are Unity titles. On Steam, roughly half of all released games use Unity. No other engine comes close for raw volume — though Unreal games tend to achieve higher individual sales.
