JavaScript
Hacking Puzzles

JavaScript was built in ten days, runs everywhere, and behaves in ways that will make you question reality. This page is your guided tour through the weirdest corners of the language — with interactive puzzles to test what you think you know.

⚡ The Ten-Day Language

In May 1995, Brendan Eich was hired by Netscape with one brief: write a scripting language for the browser in ten days. It was originally called Mocha, then LiveScript, then rebranded to JavaScript to piggyback on Java's hype — even though the two languages have almost nothing in common.

The tight deadline left permanent marks. Some design decisions were genuinely clever; others were quick compromises that couldn't later be fixed without breaking the entire web. Today JavaScript runs on billions of devices, and that original ten-day skeleton is still there underneath.

The Wat Talk. In 2012, Gary Bernhardt gave a famous four-minute conference talk — just him, a REPL, and increasingly unhinged JavaScript expressions. The crowd's laughter gets more uncomfortable with each result. The puzzles on this page draw from the same vein of delightful wrongness.

Why learn the weird stuff?

Knowing the quirks makes you a better debugger. When a bug appears because null is somehow equal to undefined, or a loop always logs the same number, you need the mental model — not just the symptom. The surprises also reveal how the language actually works under the hood, which is genuinely interesting once you stop being angry about it.

The Good Parts

First-class functions, closures, prototypal inheritance, the event loop, dynamic objects — surprisingly elegant when used intentionally.

The Wat Parts

Implicit type coercion, hoisting, this binding, floating point gotchas, typeof null — legacy decisions baked into the language forever.

🔀 Type Coercion — The Wild West

JavaScript is a dynamically typed language, which means values carry their own type information and can be freely mixed. When you combine values of different types, JavaScript doesn't throw an error — it silently coerces one or both values into a common type, using rules that feel arbitrary until you understand the algorithm behind them.

The + operator has a split personality

The + operator does two completely different things: numeric addition and string concatenation. The rule is simple but has huge consequences: if either operand is a string, the result is a string. The minus operator, by contrast, always means subtraction and always coerces both sides to numbers.

// + coerces to STRING if either side is a string "5" + 3 // → "53" "5" + 3 + 1 // → "531" (left to right) 1 + 2 + "3" // → "33" (1+2 happens first = 3, then "3"+"3") // - always coerces to NUMBER "5" - 3 // → 2 "10" - "4" // → 6

Falsy values — the only six that matter

The == rabbit hole

The == operator uses the Abstract Equality Comparison algorithm, which performs coercion before comparing. The gist: null only equals undefined (and nothing else), numbers and strings get converted to numbers, and booleans get converted to 0 or 1 first. The === operator skips all of that and requires identical type and value — and it's almost always what you want.

Rule of thumb: always use === unless you have a specific reason to use ==. The one legitimate use of == is x == null, which is a concise way to check for both null and undefined.

🔭 Scope, Hoisting & Closures

JavaScript's scoping rules were designed in a hurry and later patched. Understanding them requires knowing that the JS engine processes code in two passes: a compilation phase (where declarations are registered) and an execution phase (where code actually runs).

Hoisting: the engine reads ahead

var declarations are "hoisted" to the top of their containing function or global scope. But only the declaration is hoisted, not the initialisation. The result: you can reference a var before its line in the code without getting a ReferenceError — you just get undefined.

// What you write: console.log(x); // → undefined (not an error!) var x = 5; console.log(x); // → 5 // What the engine sees: var x; // declaration hoisted console.log(x); // → undefined x = 5; // assignment stays in place console.log(x); // → 5

let and const are also hoisted, but they live in a Temporal Dead Zone from the start of their block until their declaration line. Accessing them in this window throws a ReferenceError — a far more helpful behaviour than silently returning undefined.

The classic var-in-loop bug

This is one of the most common JavaScript gotchas and one of the best illustrations of how closures capture variable references, not values:

// The bug — using var for (var i = 0; i < 3; i++) { setTimeout(() => console.log(i), 0); } // Prints: 3, 3, 3 // All callbacks share the SAME i variable, which is 3 when they run

The loop finishes (i reaches 3) before any callback fires. All three closures point to the same i variable — and by then, it's 3. Using let fixes this instantly, because let creates a new binding for each loop iteration:

// The fix — using let for (let i = 0; i < 3; i++) { setTimeout(() => console.log(i), 0); } // Prints: 0, 1, 2 // Each iteration gets its own i — the closure captures that unique binding

What is a closure, really?

A closure is a function that "remembers" the variables from the scope in which it was created, even after that scope has returned. Every function in JavaScript is a closure — it carries a reference to its outer scope. This is what enables patterns like private state, factory functions, and memoisation.

function makeCounter() { let count = 0; return { increment: () => ++count, get: () => count, }; } const c = makeCounter(); c.increment(); // 1 c.increment(); // 2 c.get(); // 2 // count is private — accessible only through the closure
Key insight: closures capture variables by reference, not by value. When the variable changes, every closure sharing it sees the new value. This is both the source of the loop bug and the source of closures' power.

🎭 The Slippery this

In most object-oriented languages, this refers to the current instance and is determined at class definition time. In JavaScript, this is determined at call time — it depends on how a function is invoked, not where it's defined. This trips up virtually every developer who comes to JS from another language.

The four binding rules

1
Default binding. A function called without any context uses the global object as this (window in browsers). In strict mode, this is undefined instead — safer and more sensible.
2
Implicit binding. When a function is called as a method — obj.fn() — this is obj. The object to the left of the dot is the binding.
3
Explicit binding. fn.call(ctx), fn.apply(ctx), and fn.bind(ctx) let you manually set this to any value. bind creates a permanently bound new function.
4
Arrow functions. Arrow functions have no own this. They inherit this from the enclosing lexical scope at the time they're defined. This makes them ideal for callbacks.
const alice = { name: 'Alice', greetRegular: function() { return this.name; }, greetArrow: () => this.name, // 'this' is outer scope (likely window) }; alice.greetRegular(); // → "Alice" (implicit binding: alice) alice.greetArrow(); // → undefined (arrow: outer this, not alice) const fn = alice.greetRegular; fn(); // → undefined (default binding: no context) fn.call(alice); // → "Alice" (explicit binding)
The lost-context trap. Extracting a method into a variable (const fn = obj.method) strips the binding. This is why React class components used this.handleClick = this.handleClick.bind(this) in constructors — or why arrow function class fields are now preferred.

🧬 Prototypes & the Inheritance Chain

JavaScript has no classical inheritance. Instead, every object has an internal [[Prototype]] link pointing to another object. When you access a property that doesn't exist on an object, the engine walks up this chain until it finds it or reaches null. This is the prototype chain.

How new works — step by step

When you call new Constructor(), JavaScript performs four silent steps. Understanding them demystifies both new and the class syntax (which is just sugar on top):

1
Create a new empty object {}.
2
Link the object's [[Prototype]] to Constructor.prototype.
3
Invoke Constructor with this bound to the new object.
4
Return the new object — unless the constructor explicitly returns a different object.
function Animal(name) { this.name = name; // set on the instance } Animal.prototype.speak = function() { return `${this.name} speaks`; }; const cat = new Animal('Mochi'); cat.speak(); // → "Mochi speaks" // 'speak' not on cat, found on Animal.prototype via chain cat.hasOwnProperty('name'); // → true (name is on the instance) cat.hasOwnProperty('speak'); // → false (speak is on the prototype) Object.getPrototypeOf(cat) === Animal.prototype; // → true

class is sugar

ES6 class syntax is not a different inheritance system — it's a cleaner way to write constructor functions and prototype assignments. The prototype chain works exactly the same way underneath. If you've used classes in JS and never hit a prototype bug, that's fine — but knowing what's happening under the hood helps when things go wrong.

Dangerous pattern: mutating built-in prototypes (Array.prototype.myMethod = ...) used to be popular ("monkey-patching") but is now strongly discouraged. If two libraries both modify Array.prototype with the same name, one silently overwrites the other.

🌀 The Event Loop — Time is an Illusion

JavaScript is single-threaded — only one piece of code runs at a time. And yet, it handles network requests, timers, and user events concurrently without blocking. The secret is the event loop, which orchestrates three distinct queues with strict priority rules.

The execution order

The event loop follows this priority, strictly: run the entire call stack → drain all microtasks (including any microtasks added by microtasks) → run one macrotask → drain all microtasks again → repeat. This is why Promises always resolve before a setTimeout with a zero delay.

console.log('1'); // sync → runs first setTimeout(() => console.log('2'), 0); // macrotask → runs last Promise.resolve().then(() => console.log('3')); // microtask → runs second console.log('4'); // sync → runs first (after '1') // Output: 1, 4, 3, 2

Why setTimeout(fn, 0) isn't instant

setTimeout(fn, 0) doesn't mean "run immediately" — it means "put this in the macrotask queue with a minimum delay of 0ms". The function won't run until the current call stack is empty and all pending microtasks have been processed. In practice, browsers clamp the minimum to ~4ms for nested timers. It's better understood as "run this after everything else currently pending".

async/await is syntactic sugar over Promises. await suspends the current function and queues the rest as a microtask. So async functions participate in the microtask queue — they run after synchronous code, but before macrotasks.

The "never-ending microtask" trap

Because the event loop drains all microtasks before moving to the next macrotask, if microtasks continuously add more microtasks, the macrotask queue (and thus rendering and I/O events) can starve indefinitely. This is a real way to freeze a browser tab — avoid microtask loops.

🧪 Operator Weirdness & Hidden Powers

JavaScript has several operators that are either surprising, underused, or both. Some are genuine pitfalls; others are elegant shortcuts once you know them.

The comparison chaining trap

Mathematical notation like 1 < 2 < 3 feels readable, but JavaScript evaluates it left to right. 1 < 2 is true, then true < 3 coerces true to 1, so 1 < 3 is true. But 3 > 2 > 1: 3 > 2 is true, then true > 1 is 1 > 1 — false.

1 < 2 < 3 // → true (true < 3 → 1 < 3) 3 > 2 > 1 // → false (true > 1 → 1 > 1)

NaN is not equal to itself

NaN (Not-a-Number) is the only value in JavaScript that is not equal to itself: NaN === NaN is false. This is actually mandated by the IEEE 754 floating-point standard. To check for NaN, use Number.isNaN(x) — the older isNaN(x) function first coerces its argument, so isNaN("hello") is true, which is often a surprise.

NaN === NaN // → false isNaN("hello") // → true (coerces first: NaN === NaN) Number.isNaN("hello") // → false (no coercion: "hello" is a string) Number.isNaN(NaN) // → true
